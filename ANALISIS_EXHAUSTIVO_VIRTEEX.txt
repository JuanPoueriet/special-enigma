1. DOMINIO DE FACTURACIÓN (BILLING) - CRÍTICO
================================================================================

1.1. Proveedor de Timbrado (PAC) - Simulaciones y Errores de Lógica
   Archivo: libs/domains/billing/infrastructure/src/lib/providers/finkok-pac.provider.ts
   - Lógica de Cancelación Falsa (Línea 168): El método `cancel` retorna `true` incondicionalmente si recibe una respuesta estructuralmente válida de Finkok, sin validar el contenido de `cancelResult`.
     -> Impacto: El sistema marcará facturas como canceladas aunque el SAT rechace la cancelación (e.g., documentos no cancelables).
   - URLs Hardcodeadas (Línea 144): La URL de cancelación se deriva mediante reemplazo de strings (`replace('/stamp', '/cancel')`). Si la configuración de entorno cambia, esto romperá producción.
   - Namespaces Fijos (Línea 38): `@_xmlns:apps` está fijo a `http://facturacion.finkok.com/stamp`. Esto impide cambiar de entorno (Demo vs Prod) limpiamente sin tocar código.

1.2. Interfaz de Facturación (Frontend) - Cálculos y Validaciones Deficientes
   Archivo: libs/domains/billing/ui/src/lib/pages/invoices/new/new.page.ts
   - Impuestos Hardcodeados (Líneas 89, 110): Se asume un `taxRate` fijo de 0.16 (16%) para todos los productos.
     -> Impacto: Imposible facturar productos con Tasa 0%, Exentos (e.g., Medicinas, Libros) o Frontera (8%).
   - Cálculo de Impuestos Local Simplista (Línea 149): `tax += lineTotal * taxRate`. Ignora completamente IEPS, Retenciones de ISR/IVA y tasas locales.
   - Pérdida de Datos en Envío (Línea 196): El payload `CreateInvoiceDto` enviado al backend ignora la tasa de impuesto editada por el usuario en el formulario. El backend recalculará con sus propios datos, creando discrepancia entre lo que ve el usuario y lo que se factura.
   - Validación de Stock Incompleta (Línea 159): `validateStock` retorna `true` si no hay `productId` seleccionado, permitiendo líneas vacías.

   - Cuotas IMSS Fijas (Líneas 6-7): `IMSS_BASE_RATE` y `IMSS_EXCESS_RATE` fijos. No permite configuraciones patronales específicas o actualizaciones de ley.
   - Días por Periodo Fijos (Línea 85): Asume `MONTHLY = 30`, `WEEKLY = 7`. Ignora años bisiestos y meses de 31/28 días, afectando el cálculo exacto del SBC.

2.2. Estrategia Fiscal USA - Lógica No Comercial
   Archivo: libs/domains/payroll/infrastructure/src/lib/strategies/us-payroll.strategy.ts
   - Tablas de Impuestos Hardcodeadas (Líneas 34-40): Las tablas de ISR Federal 2024 están "quemadas" en el código con `if/else`.
     -> Impacto: Inmantenible. Requiere cambio de código cada año fiscal. Debe ser Database-Driven.
   - Tasas FICA Fijas (Líneas 46-50): Seguro Social y Medicare fijos.

2.3. Caso de Uso de Timbrado - Datos Falsos en XML (Mocking)
   Archivo: libs/domains/payroll/application/src/lib/use-cases/stamp-payroll.use-case.ts
   - CURP Hardcodeada (Línea 108): `payroll.employee?.curp || 'AAA010101AAA000000'`.
     -> Impacto: El SAT rechazará inmediatamente cualquier timbrado que use este fallback en producción.
   - Régimen y Contrato Fijos (Líneas 109-113): Todos los empleados se timbran como "Sueldos y Salarios" (02), "Indeterminado" (01) y "Quincenal" (04).
     -> Impacto: Ilegal para empleados eventuales, semanales, o asimilados a salarios.

   - Mapeo de Conceptos Simplista (Línea 150): Mapea conceptos basándose en strings (`includes('sueldo')`). Extremadamente frágil. Si el concepto se llama "Salario Base", funciona; si se llama "Pago Normal", falla.
